# Architecture

## Agent hierarchy

```
image_studio_root_agent.root_agent  (LlmAgent)
│
├── tools
│   ├── virtual_try_on         (calls Vertex AI virtual-try-on-001)
│   └── edit_image             (calls Vertex AI Nano Banana)
│
└── sub_agents
    └── marketing_campaign_agent  (SequentialAgent)
        │
        ├── brand_parser           (LlmAgent)
        │   ├── tool: find_vto_image          (scans state for hero photo)
        │   ├── tool: find_brand_pdf          (scans state for guidelines PDF)
        │   └── tool: extract_brand_guidelines
        │       (reads PDF via Gemini 2.5 Flash multimodal → JSON brand profile)
        │
        ├── creative_director      (LlmAgent)
        │   └── tool: save_creative_plan
        │       (writes structured plan to state['creative_plan'])
        │
        ├── asset_generator        (LlmAgent)
        │   ├── tool: get_creative_plan
        │   └── tool: generate_ad_creative
        │       (Nano Banana reframes the VTO photo into a 1200x628
        │        landscape marketing image, Pillow center-crops to spec)
        │
        └── social_media_publisher (LlmAgent)
            └── tool: generate_social_post
                (Nano Banana reframes VTO photo for Instagram 4:5 and
                 9:16 + TikTok 9:16; captions + hashtags saved as
                 text artifacts alongside the images)
```

## How control flows

The root agent is an `LlmAgent`. Its instruction tells the LLM:
"if the user says 'make ads' or similar, transfer to
`marketing_campaign_agent`." ADK's transfer mechanism passes the same
`InvocationContext` to the sub-agent, so the entire session state — every
artifact, every prior tool result — is visible.

Inside the campaign pipeline, control is **deterministic** (not LLM-routed)
because it's a `SequentialAgent`. The four sub-agents run in fixed order,
each reading from / writing to `session.state`.

## State keys

| Key | Written by | Read by | Type |
|---|---|---|---|
| `vto_image_filename` | `find_vto_image` tool | `generate_ad_creative`, `generate_social_post` | str |
| `brand_pdf_filename` | `find_brand_pdf` tool | `extract_brand_guidelines` | str |
| `brand` | `extract_brand_guidelines` tool | `creative_director` & `social_media_publisher` (instructions read it) | dict |
| `creative_plan` | `save_creative_plan` tool | `get_creative_plan` tool | dict |

## Artifacts (file objects in the session)

| Artifact | Created by | Used by |
|---|---|---|
| `<upload>` (person photo, product photo, brand PDF) | `SaveFilesAsArtifactsPlugin` on user upload | tools via `tool_context.load_artifact` |
| `vto_result_<hex>.png` | `virtual_try_on` tool | shown inline; read by `generate_ad_creative` and `generate_social_post` |
| `edit_result_<hex>.png` | `edit_image` tool | shown inline; same as above |
| `ad_marketing_image_landscape_<hex>.png` | `generate_ad_creative` | shown inline; user uploads to Google Ads |
| `social_instagram_feed_<hex>.png` + `.txt` | `generate_social_post` | image + caption shown inline |
| `social_instagram_reels_story_<hex>.png` + `.txt` | same | same |
| `social_tiktok_vertical_<hex>.png` + `.txt` | same | same |

## Output spec mapping

| Asset | Aspect | Pixel size | Source |
|---|---|---|---|
| Google Ads — Marketing image landscape | 1.91:1 | 1200 × 628 | Nano Banana edits VTO photo |
| Instagram — feed post | 4:5 | 1080 × 1350 | Nano Banana edits VTO photo |
| Instagram — Reels / Story | 9:16 | 1080 × 1920 | Nano Banana edits VTO photo |
| TikTok — vertical | 9:16 | 1080 × 1920 | Nano Banana edits VTO photo |

Google Ads also requires square (1:1) marketing images and logo assets for
a full Responsive Display campaign — this build skips those because
Imagen rejects prompts that resemble brand-logo cloning, and the user is
better off uploading their real square images and logo directly to
Google Ads.

All assets emit PNG (JPEG fallback if PNG > 5 MB). Center-cropped to
exact aspect ratio with Pillow before final resize.

## Why one image model?

**Nano Banana (`gemini-2.5-flash-image`)** is image-in, image-out
editing. Given the VTO photo as input, it preserves the person's face
and the product geometry while restyling the scene (background, lighting,
framing) on-brand. Every output asset in this build is a recomposition
of the same VTO photo for a different aspect ratio, so Nano Banana
handles all four. Imagen 4 is still imported by the codebase for use if
you re-enable logo or text-to-image assets later.

## Why not put text on the images?

Google Ads applies headlines / descriptions as overlays at serve time. If
you bake heavy text into the pixels, the asset is also evaluated for ad
strength as if it were a static image ad — and Google explicitly penalises
heavy in-image text on Responsive Display assets. The creative briefs
generated by `creative_director` instruct the image models to leave clear
space on one side for the overlay instead.

## Deployment surface

One `agent_engines.create()` call deploys the **entire** tree as a single
reasoning engine. There's one resource ID, one billing line, and one
endpoint. Internal agent-to-agent transfers happen inside the engine
container — no network hops between agents.

The deploy script bundles all three sibling packages (`vto_agent/`,
`marketing_campaign_agent/`, `image_studio_root_agent/`) via `extra_packages` so
the cloudpickled `root_agent` can resolve its imports at runtime.

## Pillow note

Image-generation models honour *aspect ratios*, not exact pixel counts.
The `_fit_to_size` helper in `marketing_campaign_agent/agent.py`:

1. Center-crops the generated image to the exact target ratio (1.91:1,
   1:1, 4:1).
2. Resizes to the recommended Google pixel count using Lanczos.
3. Falls back to JPEG @ 92 if PNG would exceed 5 MB.

This is why Pillow is in the runtime requirements list.
